机会与痛点

最有价值的数据，恰恰是最被"锁住"的数据

药企、医院手握全球最高价值的数据资产，却因监管、隐私与确权难题而无法流动。这正是 Bio-AI 的核心瓶颈，也是 SETU 的差异化战场。

今天为什么做不到

  • 子网只是"带隔离的代币命名空间"，执行语义与主链相同——没有数据护照、没有可编程授权
  • 没有"非转账类"的机密计算能力，数据无法在不泄露下被使用
  • 没有可信回执与证据图，贡献无法被精确确权
  • 没有协议级托管、计量与自动分账，付款无法可信地折现给贡献方
  • 合约层够不着闭环四要素：自主触发、可信证明可见、跨域原子写、主权隔离

本方案带来什么

  • 数据护照：机构登记数据/模型资产，链上只放"指纹"与授权策略，原始数据永不离开机构
  • 可编程授权：精确控制"谁、何时、能跑什么计算、跑几次、分多少钱"，撤销即时生效
  • 机密计算：数据仅在可信执行环境内短暂解密、用完即焚，执行者拿不走原文
  • 可验证回执：每次计算产出带签名的回执，进入不可篡改的证据图
  • 自动结算：真实付款进协议托管，里程碑达成后原子、可信地自动分账
一句话方案

分层而治：钱、授权、账本各归其位

把不同信任等级的关注点放到对的位置，再用"单事件、多子网、原子写"把它们安全地缝合在一起。

🛂

数据层 Data Layer

落地于 · 机构数据子网

数据/模型护照 + 可编程授权（Grant / Revoke）。机构域既是审计边界，也是主权隔离域。

协议托管 + 计量 + 自动分账。只记账、不存钱——是公共审计账本，不是公共钱包。

资金与身份

落地于 · ROOT 主链

SETU 资金、用户身份、防重放标记留在主链。钱在这里发生移动，信任不外包。

系统架构

三域协同：钱在 ROOT，账本在 SETTLEMENT，主权在机构

三个域各司其职，一笔结算事件可原子地横跨三域——全有或全无，由统一冲突预检保证，无需复杂的两阶段提交。

从"数据可用不可见"到"贡献可计可现"，每一步都由协议原生保证，而非依赖人工或合约。

🛂

数据护照与授权

DATA LAYER

机构以护照登记数据资产，以授权精确控权。

  • 原始数据永不上链，仅承诺 + 策略
  • 精确控制谁 / 何时 / 跑什么 / 跑几次 / 分多少
  • "撤销即斩"——下一轮共识后对所有新任务即时失效
🔒

机密计算与回执

COMPUTE

计算在可信执行环境内进行，产出可验证回执。

  • 执行前完成授权校验（未撤销 / 在有效期 / 类型允许 / 未超次）
  • 数据仅在加密飞地短暂解密，用完即焚
  • 回执含模型/数据版本、环境指纹、指标，进入证据图
💸

协议托管与自动分账

ECONOMIC LAYER

真实付款进托管，里程碑达成自动结算。

  • 资金由"规则"保管，而非私钥
  • 单事件原子完成扣托管 + 多方入账 + 记账
  • 重复触发不可能双花，过期资金必可退款
端到端业务流程

一个真实场景：药企采购 → 自动分账

以"药企付费、医院授权数据、研究者跑模型、实验室反馈、贡献方自动拿钱"为例，看协议如何把整条价值链自动串起来。

1

药企下单付费

药企提交带 SETU 的采购订单，并快照当时的分润策略。资金进入协议托管。

原子：ROOT 扣款→托管入账 ｜ SETTLEMENT 创建托管记录
2

医院颁发授权

医院针对自己的数据护照颁发授权，精确限定计算类型、次数与分成。

数据子网：创建 Grant 授权对象
3

研究者发起计算

任务准备阶段把授权、护照连同其证明随任务传入；可信环境内校验授权→解密→执行→封装回执→销毁明文。计算者全程零数据 I/O。

原子：数据子网写入回执 + 推进调用计数/版本
4

可信验证入图

验证器校验远程证明，并绑定回执引用的护照/授权版本，确认无误后进入证据图并被共识终结。

5

实验室反馈闭环

湿实验仪器以签名提交反馈，验签通过后证据边闭合——线上线下证据链打通。

6

自动分账

结算执行器扫描到满足里程碑的证据，自动构造分账事件，原子地把托管资金分给所有贡献方。

原子：ROOT 扣托管→多方入账 ｜ SETTLEMENT 推进+分账记录
7

计数滚动，循环可持续

下一轮计算读到新的计数与版本，形成可持续、可审计的协作经济循环。

信任与安全

资金安全：由数学与规则保证，而非靠"别出错"

面向监管与资金的系统，安全性必须是结构性的。本方案用多道独立防线，让"凭空造币"和"双花"在工程上不可能发生。

🔑资金由规则保管，非私钥

托管地址是无对应私钥的系统地址——资金只能被合法的结算/退款事件按规则动用，没有任何人能"签字"挪走它。

🧮守恒原语：永远 Σ入账 = 扣减

分账的最后一道原语原生强制资金守恒与幂等。即使上层分润策略将来出 bug，最多错配比例，永远无法凭空造币或双花托管。

🔁验证即执行

结算是公开的确定性逻辑——每个验证者在事件入账前都重新计算一遍，结果一致才接受。幂等锚保证同一里程碑全网至多一笔分账。

🛡️信任根：可信证明 + 可重跑挑战

机密计算的信任来自 TEE 远程证明（证明跑的是规范代码）；任何状态变更无有效证明则不被采信，且结果可被重跑挑战。

技术路线裁决

为什么是协议原生，而不是智能合约

把每种逻辑放到对的信任层。系统级正确性（动钱、授权）必须原生保证；灵活性留给上层，但绝不让它危及资金安全。

① 协议不变量
授权语义、结算语义、资金守恒 → 统一原生执行器：全网一套规范代码，可审计、可形式化、重点安全审计。
② 机构策略
允许的计算类型、调用次数、分成比例 → 声明式数据：由统一代码确定性解释。机构差异化靠"策略"，不靠各写各的代码。
③ 应用级逻辑
未来图灵完备的扩展逻辑（若需）→ MoveVM 合约：沙箱运行、开发者自负安全，作为闭环建成后的生态扩展。
"资金守恒不能取决于某个合约作者是否写对一句 assert——系统级正确性必须原生。"

边界与风险

清晰的本期边界，诚实的风险披露

本期聚焦闭环核心，以下能力明确不在范围内（留待生态扩展或后续迭代）：

不做 gas / 费用模型 不做 子网币 ↔ 子网币互换 不做 图灵完备分润策略 不做 合约层集成 不做 ZK / MPC 隐私计算 不做 主观贡献计价
主要风险缓解策略
资金路径出错 = 资金事故规则极简（仅分账/退款可动托管币）+ 入口拒绝清单 + 对象属主级不变量 + 双守恒/幂等测试 + 上线前安全审计
"撤销即斩"对业务方反直觉（在途任务作废）接口明确返回执行结果；对外定位为"合规卖点"而非缺陷
链上元数据泄露（暴露商业关系）本期接受并向客户披露；后续方向：敏感字段承诺化、指标聚合粗化、远期选择性披露
机构既当裁判又当选手（控镜像白名单 + 收分账）买方 opt-in 钉扎可接受镜像集 + 白名单变更走治理流程，机构不能单方面热改
