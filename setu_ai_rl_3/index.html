warning

The Missing Context Problem

AI/RL 不缺结果，缺的是结果背后的可信上下文。一个跑分背后，模型版本、数据版本、评测脚本、seed、运行环境和输出摘要往往不可验证；一个主观判断背后，谁质疑、谁修订、为什么被接受，也很难追踪。

Setu 解决的不是“结果上链”，而是把 AI 工作中的执行、判断、修订和贡献结算组织成可验证的因果历史。

> Deliberation process captured

The Four Pillars of Setu

PILLAR 01

Verified Benchmark

以 Solver 执行和 TEE 执行证明作为生产级目标路径，把自报式跑分转化为可审计、可复核、可引用的 benchmark receipt。

PILLAR 02

Trusted Provenance

将模型和数据的身份、版本、哈希与收益分配逻辑记录在链上，同时把真实权重和数据集保留在链下，以兼顾隐私、成本和可扩展性。

PILLAR 03

Verified Deliberation

记录一个判断如何被提出、质疑、修订和支持。它不裁决观点真假，而是确认审议过程中的因果历史，让一个判断在多主体挑战下逐步积累鲁棒性。它不替代同行评审或治理裁决，只提供可追踪、抗篡改的审议因果记录。

PILLAR 04

Solver Market

Solver 节点执行 AI/RL 任务，未来可通过 TEE remote attestation + measurement allowlist 验证执行环境，目标是通过 Setu 原生结算自动分配任务费用和贡献收益。

THE MAIN LOOP · DIRECTIONAL

Verified Optimization Loop
真实场景里的端到端可信优化（目标形态）

前面四个支柱不是孤立的能力，而是同一条主线的环节。作为目标形态，benchmark 不应是一次性的离线跑分，而应是一个持续过程：让 policy 在受控、可版本化的真实任务环境中 rollout，由客观指标衡量，再引入人和 Agent 的主观判断进行解释；据此修订 policy、模型或任务规格，更新后的 policy 再回到 rollout，形成闭环。Setu 的方向是用 logic clock / 时序逻辑把这条闭环串成可验证的因果历史：什么必须先于什么，谁的判断基于谁的结果。

这条闭环本质就是 RL 的 rollout → reward → policy update 循环。不同的是，Setu 把其中的 reward 拆成可验证的客观指标（Measure）+ 可追踪的主观判断（Judge），而不是一个不可解释的标量。前面把工作二分为客观 / 主观，是为了分别处理；在闭环里两者再合成一个可解释的 reward。这正是 human-in-the-loop RL / RLHF 的可信化。

play_circle

01 · Rollout

Solver 让 policy 在受控、可版本化的真实任务环境中持续 rollout；policy、模型与数据版本由 passport 锚定，输入与环境被提交。

monitoring

02 · Measure

基于客观 reward / 指标（成功率、回报、延迟等）生成可审计、可复核的 receipt；TEE attestation 接入后升级为硬件级 verified receipt。

rate_review

03 · Judge

人与 Agent 的主观判断介入：质疑、支持、解释标量 reward 背后的语境。

tune

04 · Update

基于客观 reward 与主观判断更新 policy、模型或任务规格。

replay

05 · Re-run & Settle

更新后的 policy 回到 rollout；每轮贡献按 receipt 自动结算，形成可追踪、可复核的优化迭代。

rollout → measure → judge → update → re-run → settle · RL loop ordered by logic clock / temporal logic

Where This Matters / 典型场景

Verified Benchmark for Papers
论文与打榜的可信跑分

研究者提交模型版本、数据版本、评测脚本、seed、环境和输出摘要，生成可审计、可复核的 benchmark receipt；TEE attestation 成熟后升级为硬件级 verified receipt，可放进 paper、OpenReview、GitHub、HuggingFace 和 leaderboard。

Private Test Set Evaluation
私有测试集评测

数据方不公开 hidden test set，模型方提交模型或推理任务，Solver/TEE 在受控环境中执行，目标是只输出可审计分数和 receipt；TEE attestation 成熟后提供硬件级可信分数。适合医学、金融、法律和企业内部 agent benchmark。

Agent / Research Deliberation
审议与判断演化

人和 Agent 对论文、设计方案或战略判断进行 critique、revision、support。Setu 记录谁基于谁、谁反驳谁、谁修订谁，形成可追踪、可复核的判断因果历史。

Benchmark Bounty & Settlement
任务悬赏与自动结算

组织方发布 benchmark 或任务悬赏，Solver 执行任务，结果确认后自动结算给模型方、数据方、评测维护方和算力提供方。

Trusted RLHF / Policy Eval
可信 RLHF 与受控真实任务环境策略评估

数据方提供受控、可版本化的任务环境，Solver 跑 policy 的 rollout，人给出偏好 / 奖励判断，客观 reward 与主观判断全程因果可追。适合 RLHF、agentic RL 和真实场景的策略迭代与复现。

Beyond a Ledger: A Trusted Experimental Machine

普通链可以记录 hash，但很难证明任务是否按指定规则执行，也很难追踪判断如何被挑战和修订。Setu 的机会在于，把 DAG-BFT finality、Solver execution、logic-clock causality 和 native settlement 放进同一个工作流。因此 Setu 不只是可信公告栏，而是可信实验机：证明任务如何执行、判断如何形成、贡献如何结算。

Setu 不训练模型、不托管大文件、不裁决主观对错；它做的是执行证明、因果记录和贡献结算。

说明：receipt 的可信强度分阶段演进。第一阶段提供可追踪、可复核的因果记录；后续接入 TEE remote attestation 与 measurement allowlist 后，升级为硬件级执行证明。

Path: auditable receipts → model/data passport → private-set eval → solver settlement → controlled real-world optimization loop, with verified deliberation as the long-term thesis.
